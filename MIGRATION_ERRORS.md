- [x] **P0.1 — JDK trust store rejects Maven Central's TLS chain.** *(fixed 2026-05-30, local-machine only.)*
  - First `mvn -B clean verify` failed with
    `PKIX path building failed: ... unable to find valid certification path to requested target`
    while fetching `tycho-build-5.0.3.pom` from `repo.maven.apache.org`.
    Adoptium `jdk-21.0.11.10-hotspot` bundled `cacerts` did not trust the cert
    chain that endpoint is currently serving on this machine.
  - **Fix:** added `.mvn/jvm.config` with
    `-Djavax.net.ssl.trustStoreType=Windows-ROOT` and
    `-Djavax.net.ssl.trustStore=NUL` so Maven's JVM reads roots from the
    Windows certificate store. `.mvn/jvm.config` is loaded before any
    extension, so the Tycho pomless bootstrap and all subsequent Aether HTTP
    calls now succeed without any environment variable.
  - **Why not committed:** `Windows-ROOT` is a Windows-only `KeyStore` type
    (SunMSCAPI). Forcing it from a checked-in file would break Linux CI
    runners. `.mvn/jvm.config` is therefore listed in
    [.gitignore](.gitignore) and stays as a per-developer local override.
  - **For Linux CI (when wired up):** rely on Adoptium's default `cacerts`
    or run `update-ca-certificates` in the runner image. Do NOT copy
    `Windows-ROOT` into the CI config.
  - **Verification:** with `MAVEN_OPTS` unset, `mvn -B clean verify` now
    advances to `[INFO] Scanning for projects...` with no PKIX exception. The
    next failure is P0.2.

- [x] **P0.2 — Tycho 5.0.3 extension loads but does not register the
      `eclipse-plugin` / `eclipse-feature` / `eclipse-repository` packaging types.** *(fixed 2026-05-30.)*
  - Once TLS was bypassed, all Tycho 5.0.3 jars downloaded successfully
    (`tycho-build`, `tycho-pomless`, `tycho-spi`, `tycho-api`, `p2-maven-plugin`,
    `tycho-bndlib`, `org.eclipse.jdt.core 3.45.0`, `ecj 3.45.0`, etc.), and on
    the next run Maven proceeded straight to "Scanning for projects".
  - The polyglot extension is firing — it produced virtual POMs for all three
    modules (`.polyglot.META-INF`, `.polyglot.feature.xml`,
    `.polyglot.category.xml`). The Tycho graph builder is also wired in
    (stack trace shows `org.eclipse.tycho.build.TychoGraphBuilder.build`).
  - But the build fails immediately with:
    ```
    [ERROR] Unknown packaging: eclipse-plugin     @ seg.jUCMNav/.polyglot.META-INF
    [ERROR] Unknown packaging: eclipse-feature    @ seg.jUCMNav.feature/.polyglot.feature.xml
    [ERROR] Unknown packaging: eclipse-repository @ seg.jUCMNav.repository/.polyglot.category.xml
    [ERROR] The build could not read 3 projects
    ```
  - Tycho 5.0.3 was published on 2026-05-29 (one day before this run). It is
    plausible that this is a freshly-shipped extension issue (lifecycle mapping
    not registered, missing `tycho-maven-plugin` declaration as a build plugin
    with `<extensions>true</extensions>`, or similar). The CLAUDE.md targets pin
    Tycho to 5.0.3 — **do not** downgrade without asking the user first.
  - Reproduce:
    ```
    cd C:\Users\jucmn\Claude\jUCMNav2026\jUCMNavPlus
    $env:MAVEN_OPTS = "-Djavax.net.ssl.trustStoreType=Windows-ROOT -Djavax.net.ssl.trustStore=NUL"
    mvn -B -e clean verify
    ```
  - Full log: `build-output.log` at the repo root (also produced this report).
  - **Root cause:** in Tycho 5.0.x, the `tycho-build` extension only registers
    the polyglot `Mapping` components (parsing `MANIFEST.MF`, `feature.xml`,
    `category.xml` into virtual POMs). The Maven `LifecycleMapping` components
    that actually bind `eclipse-plugin` / `eclipse-feature` / `eclipse-repository`
    packaging to Tycho's mojos live in `tycho-maven-plugin`'s
    `META-INF/plexus/components.xml`. Tycho 4.x bundled both in one extension;
    Tycho 5.0.x's `tycho-build` pom no longer depends on `tycho-maven-plugin`,
    so the lifecycle mappings are missing unless declared separately.
  - **Fix:** declare `tycho-maven-plugin` as a build *plugin* in
    [pom.xml](pom.xml) with `<extensions>true</extensions>`, NOT as a core
    extension in [.mvn/extensions.xml](.mvn/extensions.xml). Loading it as a
    core extension fails Guice DI with
    `No implementation for LegacySupport was bound while locating OsgiBundleProject ... TychoProject annotated with @Named("eclipse-plugin")`
    because the core-extension class realm cannot see
    `org.apache.maven.plugin.LegacySupport`.
  - **Verification:** `mvn -B clean verify` now resolves all packaging types,
    schedules the full reactor (parent → seg.jUCMNav → feature → repository),
    downloads the 2026-03 p2 content, and proceeds into the
    `tycho-packaging-plugin` validate phase on `seg.jUCMNav`. The next failure
    is a content-level mismatch (see P0.3 below), not a packaging-registration
    failure.

- [x] **P0.3 — Bundle-Version / Maven version mismatch on `seg.jUCMNav`.** *(fixed 2026-05-30.)*
  - First error reached after P0.2 cleared:
    `OSGi version 9.0.0 in META-INF/MANIFEST.MF does not match Maven version 9.0 in pom.xml`
    raised by `tycho-packaging:5.0.3:validate-version`.
  - Manifest declared `Bundle-Version: 9.0` (Eclipse accepts a 2-segment OSGi
    version and normalizes it to `9.0.0`, but Tycho's pomless extension
    derived Maven `version=9.0` verbatim from that string, and the validator
    then refused the `9.0.0` vs `9.0` mismatch).
  - **Fix:** changed
    [seg.jUCMNav/META-INF/MANIFEST.MF](seg.jUCMNav/META-INF/MANIFEST.MF) line 5
    from `Bundle-Version: 9.0` to `Bundle-Version: 9.0.0.qualifier`. The
    `.qualifier` literal lets Tycho substitute a timestamp/SCM qualifier at
    build time; the pomless extension now derives Maven `version=9.0.0-SNAPSHOT`
    and Tycho accepts it. Reactor summary confirms:
    `[bundle] jUCMNav 9.0.0-SNAPSHOT`.
  - **Secondary issue uncovered:** with the manifest edit (and the P0.1/P0.2
    changes) sitting uncommitted, `tycho-buildtimestamp-jgit` refused to
    compute a qualifier with `Working tree is dirty`. Tightened
    [pom.xml](pom.xml) `tycho-packaging-plugin` config with
    `<jgit.dirtyWorkingTree>warning</jgit.dirtyWorkingTree>` so the build
    falls back to the build-time timestamp provider whenever the tree is
    dirty. CI on a clean checkout still gets the strict git-based qualifier;
    iterative migration work no longer has to commit after every edit.
  - **Verification:** `mvn -B clean verify` now passes the entire Tycho
    preflight (project scanning, target-platform resolution, `clean`,
    `build-qualifier`, `validate-id`, `validate-version`) and reaches the
    `tycho-compiler-plugin:5.0.3:compile` phase on `seg.jUCMNav`. The
    compiler emits **362 real Java errors** against modern Eclipse / EMF /
    OCL / GEF / Workbench APIs. Sections 1–7 below are now ready to be
    populated from this output.

       picked them up via PDE classpath resolution even after they were
       dropped from Bundle-ClassPath.

       `org.eclipse.orbit.xml-apis-ext` transitively (via Batik / xmlgraphics

    [UrnTableContentProvider.java](seg.jUCMNav/src/seg/jUCMNav/views/search/UrnTableContentProvider.java) and

    `org.eclipse.search.internal.ui.text.IFileSearchContentProvider`.

    - Table impl: count the result's elements, capped at the page's
      `elementLimit`.
    - Tree impl: recursive leaf walk — for each child, if `hasChildren`
      then recurse, else `count++`.
  - **No semantic change:** these methods are only consulted by the
    Search view to render result counts; existing behavior preserved
    because we copied upstream semantics verbatim.
  - **Verification:** error count 21 → 19. Zero remaining errors in
    `views/search/`.
